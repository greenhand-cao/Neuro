# 第一部分：产品愿景（Vision）

## 为什么世界需要 Neuro？

当前的知识管理工具存在根本性分裂：

- **Zotero / Mendeley** — 擅长文献管理，但缺乏 AI 分析与知识关联
- **Obsidian / Notion** — 擅长笔记与知识链接，但缺乏自动化的信息采集与 AI 驱动
- **Trello / Asana** — 擅长项目管理，但缺乏知识沉淀与智能分析
- **AI Chat（ChatGPT / Claude）** — 擅长对话与生成，但缺乏结构化的知识管理与持久化

**Neuro 的目标：** 打破这些工具之间的壁垒，创建一个统一的 **AI Native Knowledge Operating System**。

## 产品哲学

```
Everything is a Knowledge Unit.    — 一切皆知识单元
Everything has AI.                  — 一切皆 AI
Everything is Connected.            — 一切皆关联
Everything can Grow.                — 一切皆可成长
Everything can Evolve.              — 一切皆可演化
```

> **Neuro 不是一个知识管理工具（Knowledge Management System）。**
>
> **Neuro 是一个知识培育系统（Knowledge Cultivation System）。**

它不是用来"保存"知识的，而是用来"培养"知识的。

## 第一章 核心概念（Core Concepts）

### 1.1 Project（项目）

Project 是 Neuro 中最顶层的组织单位，代表一个独立的研究方向、工作领域或生活计划。

**示例：**
- `脑类器官神经电极`
- `2026 欧洲自由行`
- `创业项目 BP 准备`

### 1.2 Collection（集合）

Collection 是 Project 内部的知识分类层级，用于将相关的 Knowledge Unit 组织在一起。

**示例（在"脑类器官神经电极"项目下）：**
- `类器官培养`
- `神经电极制备`
- `多模态检测`
- `信号采集与分析`

### 1.3 Knowledge Unit（知识单元）

**这是 Neuro 的第一性原则（First Principle）。**

Neuro 管理的不是文件，不是 PDF，不是网页链接，而是 **Knowledge Unit**。

```
论文不是 PDF，论文是 Knowledge Unit。
实验不是 Excel，实验是 Knowledge Unit。
任务不是 Todo，任务也是 Knowledge Unit。
网页不是 URL，网页是 Knowledge Unit。
视频不是 MP4，视频是 Knowledge Unit。
```

#### Knowledge Unit 的数据结构

每一个 Knowledge Unit 在磁盘上对应一个文件夹：

```
Knowledge Unit: 论文1
│
├── Knowledge.md          ← AI 知识索引（大脑）
├── metadata.json         ← 结构化元数据
├── Assets/               ← 附件资源
│   ├── PDF/
│   ├── 中文PDF/
│   ├── Notes/
│   └── Website/
└── Relations             ← 关联关系（指向其他 Knowledge Unit）
```

### 1.4 Knowledge.md

**Knowledge.md 不是笔记。**

它是 Knowledge Unit 的 **AI Knowledge Index**（AI 知识索引），由 AI 持续维护和更新。

包含内容：
- 条目类型（论文/网页/视频/实验/任务等）
- 条目名称
- 作者/来源
- 摘要与关键信息
- 标签（用于可视化和知识图谱）
- 笔记与用户批注
- 附件跳转链接
- 关联关系描述
- AI 提取的结构化参数（如电极阻抗、材料等）

### 1.5 Relation（关联关系）

Relation 是 Knowledge Unit 之间的连接，构成 Knowledge Graph。

**示例：**
```
论文 A
  ↓ 引用
论文 B
  ↓ 支持
实验 C
  ↓ 产生
结果 D
  ↓ 触发
任务 E
```

> **Graph 不是图片。Graph 就是数据库。**

### 1.6 View（视图）

View 是 Knowledge Unit 的可视化呈现方式。统一称为 View，不再使用"模式"（Mode）的叫法。

支持的 View 类型（预留插件扩展接口）：
```
Markdown   — 完整文档视图
Tree       — 知识树视图
Graph      — 知识图谱视图
Timeline   — 时间线视图
Kanban     — 看板视图
Calendar   — 日历视图
Table      — 表格视图
Gallery    — 卡片视图
Mind Map   — 思维导图视图
```

```
├────────────┬──────────────────────────┬──────────────┤
│            │                          │              │

│ Tree       │     Workspace            │    Panel     │

├────────────┴──────────────────────────┴──────────────┤
│ Status Bar（状态栏）                                   │
└──────────────────────────────────────────────────────┘
```

### 2.2 Knowledge Tree（左栏）

**不再叫 Explorer 或文件树。**

Knowledge Tree 展示的是 Knowledge Tree，不是文件目录树。它展示的是 Project → Collection → Knowledge Unit 的层级结构。

### 2.3 Workspace（中栏）

主工作区，根据当前选中的 View 类型展示不同的内容：
- 选中 Markdown View → 显示 Knowledge.md 内容
- 选中 Graph View → 显示知识图谱
- 选中 Kanban View → 显示任务看板
- 选中 Timeline View → 显示时间线

### 2.4 AI Panel（右栏）

统一 AI 交互入口。不再区分"AI 对话"和"Thinking"——所有 AI 交互都在右栏完成，包括：
- 自然语言问答
- AI 对 Knowledge Unit 的分析
- AI 对 Knowledge.md 的修改指令

### 2.5 Toolbar（顶部工具栏）

- 项目切换
- View 类型切换
- 搜索
- 插件管理
- 全局设置

### 2.6 Status Bar（底部状态栏）

- 当前 Knowledge Unit 数量
- AI 状态（分析中 / 就绪）
- 同步状态

## 第四章 Knowledge Unit 生命周期（Knowledge Lifecycle）

> **这是 Neuro 最大的创新点，也是整个产品的灵魂。**
>
> 传统工具：导入 → 保存 → 结束。
>
> Neuro：知识是活的，它在不断成长。

```
① Create（创建）
   Connector 导入 / 手动上传 / 模板创建
   ↓

② Understand（理解）
   AI 读取知识内容，生成 Knowledge.md 初始版本
   ↓

③ Grow（成长）
   AI 持续完善 Knowledge.md：
     - 提取关键参数
     - 自动打标签
     - 生成摘要
     - 视频关键帧截图
   ↓

④ Connect（连接）
   建立与其他 Knowledge Unit 之间的 Relation
   用户在 Graph View 中可视化查看关联网络
   ↓

⑤ Work（工作）
   各种 View 围绕 Knowledge Unit 协同工作：
     - 在 Graph 中查看知识关联
     - 在 Kanban 中跟踪任务进度
     - 在 Timeline 中查看时间线
     - AI Panel 中回答问题
   ↓

⑥ Evolve（演化）
   随着用户的阅读、实验、写作，以及 AI 的持续推理：
     - Knowledge.md 持续更新
     - Graph 持续更新
     - 新的 Relation 不断建立
     - 知识在不断生长
```

> **Neuro 不保存知识，Neuro 培养知识。**

## 第五章 AI 引擎（AI Engine）

### 5.1 AI 的核心工作方式

AI 始终围绕 **Knowledge Unit** 工作，不是围绕 PDF，不是围绕文件。

### 5.2 AI 能力矩阵

```
Understand（理解）
   阅读 Knowledge.md，理解 Knowledge Unit 的内容与上下文

Extract（提取）
   从附件（PDF/网页/视频）中提取关键信息、参数、数据

Summarize（摘要）
   生成摘要、核心观点、关键结论

Tag（打标）
   自动生成标签，建立分类体系

Reason（推理）
   跨 Knowledge Unit 的推理分析：对比、关联、推断

Generate（生成）
   更新 Knowledge.md、生成可视化数据、生成报告

Observe（观察）
   Follow 模式下持续观察用户进度，更新任务状态

Reflect（反思）
   根据用户反馈和新的知识输入，主动优化已有 Knowledge.md
```

### 5.3 AI 访问策略

1. **本地优先**：AI 首先阅读 Knowledge.md（这是每个 Knowledge Unit 的总结性文件）
2. **按需扩展**：如需更多信息，AI 通过附件跳转链接进一步查询
3. **Token 优化**：如无必要，AI 不会主动打开附件文件
4. **用户可控**：用户可指定 AI 分析的深度与范围

### 5.4 AI 接入方式

- 支持接入多种 AI 模型（Claude / GPT / Gemini 等）
- 所有 AI 均通过统一的 AI Engine 接口调用
- 未来支持本地模型离线运行

## 第六章 关联系统（Relation System）

### 6.1 关系类型

| 关系类型 | 语义 | 示例 |
|---------|------|------|
| 引用（Cite） | A 引用 B | 论文1 引用 论文2 |
| 支持（Support） | A 支持 B | 实验数据 支持 研究结论 |
| 产生（Produce） | A 产生 B | 实验 产生 结果 |
| 触发（Trigger） | A 触发 B | 任务完成 触发 新任务 |
| 属于（BelongTo） | A 属于 B | 论文 属于 类器官培养 Collection |
| 依赖（DependsOn） | A 依赖 B | 任务B 依赖 任务A |
| 关联（RelatedTo） | A 与 B 相关 | 网页攻略1 关联 网页攻略2 |

### 6.2 关系的数据存储

- 关系不是视图层面的装饰，而是存储在数据库中的一等公民
- 每个 Relation 记录源 Knowledge Unit、目标 Knowledge Unit、关系类型
- Graph View 只是 Relation 数据的可视化呈现

### 6.3 关系推荐

- AI 根据 Knowledge.md 中的标签、关键词、引用等信息，自动推荐潜在的关系
- 用户可手动确认或拒绝推荐的关系

## 第七章 视图系统（View System）

### 7.1 统一视图框架

不再有"Entry 模式""Thinking 模式""Print 模式""Follow 模式"。

所有可视化呈现统一称为 **View**，用户通过 View 切换器在不同视图之间切换。

### 7.2 内置视图

| 视图 | 功能 | 适用场景 |
|------|------|---------|
| Markdown View | 完整渲染 Knowledge.md 内容，支持编辑 | 阅读与编辑 Knowledge Unit |
| Tree View | 展示 Project → Collection → Knowledge Unit 的层级结构 | 浏览项目结构 |
| Graph View | 以节点-边形式展示 Knowledge Unit 之间的关联关系 | 知识关联探索 |
| Timeline View | 按时间线排列 Knowledge Unit | 研究历程、旅行计划 |
| Kanban View | 看板视图展示任务状态（待办/进行中/已完成） | 任务管理 |
| Calendar View | 日历视图展示任务与事件 | 日程管理 |
| Table View | 表格化展示所有 Knowledge Unit 及其关键字段 | 批量浏览与筛选 |
| Gallery View | 卡片墙形式展示 Knowledge Unit | 视觉化浏览 |
| Mind Map View | 思维导图形式展示知识结构 | 头脑风暴与知识拆解 |

### 7.3 预留接口

- 开放插件接口，允许社区开发新的 View 类型
- 用户可以自定义 View 的布局与样式

## 第八章 模板系统（Template System）

### 8.1 模板的作用

用户创建一个 Knowledge Unit 时，选择 Template 来确定其结构和初始内容。

### 8.2 内置模板

| 模板类型 | 初始 Knowledge.md 结构 | 适用场景 |
|---------|----------------------|---------|
| Paper Template | 标题、作者、摘要、DOI、关键词、发表日期、期刊 | 学术文献管理 |
| Experiment Template | 实验名称、目的、方法、材料、参数、结果 | 科研实验记录 |
| Task Template | 任务名称、负责人、前置任务、起止时间、重要性、进度、核验 | 项目任务管理 |
| Webpage Template | 标题、URL、访问时间、摘要、截图 | 网页信息保存 |
| Video Template | 标题、来源、时长、关键帧、摘要 | 视频资料管理 |
| Meeting Template | 会议主题、时间、参与者、议程、要点、待办 | 会议记录 |
| Note Template | 标题、内容、标签 | 自由笔记 |

### 8.3 模板的可扩展性

- 用户可自定义模板
- 开源社区可通过插件贡献新模板
- AI 可根据 Knowledge Unit 内容推荐合适的模板

## 第九章 插件架构（Plugin Architecture）

### 9.1 插件化设计原则

> **Connector 不是系统功能，是 Plugin。**
>
> **Zotero 集成不是系统功能，是 Plugin。**
>
> **Obsidian 同步不是系统功能，是 Plugin。**

外部集成全部以 Plugin 形式接入，不写死在系统核心中。

### 9.2 内置插件类型

| 插件类型 | 功能 | 备注 |
|---------|------|------|
| Neuro Connector | 浏览器插件，抓取网页/文献/视频信息 | 核心插件，默认安装 |
| Zotero Sync | 从 Zotero 导入条目 | 可选 |
| Obsidian Sync | 与 Obsidian Vault 同步 | 可选 |
| Notion Sync | 与 Notion 数据库同步 | 可选 |
| Claude Code Integration | Claude Code 命令行集成 | 可选 |
| Git Integration | Git 版本控制集成 | 可选 |

### 9.3 Connector 插件功能详述

**浏览器支持：** Chrome / Firefox / Edge

**核心功能：**
1. 自动识别当前页面类型（网页/学术文献/视频）
2. 一键抓取标题、摘要、URL、封面图等元数据
3. 自动下载附件（PDF 等）
4. 保存时弹出表单，允许用户编辑条目名称、标签、简记
5. 自动归类到指定的 Collection

**使用场景：**
- 研究者浏览学术网站时一键保存文献
- 学生收集资料时快速保存网页
- 旅行规划时保存攻略和景点信息

# 第三部分：PRD 功能概述（基于 PAD 落地的功能清单）

> 以下功能清单是 PAD 的实现。每个功能的详细 PRD（含用户故事、交互流程、异常处理、验收标准）将在后续独立文档中展开。

## F1 Connector（浏览器插件）

- **目标**：实现从浏览器到 Neuro 的一键知识采集
- **核心流程**：浏览网页 → 点击插件 → 识别类型 → 抓取元数据 → 保存到指定 Project/Collection
- **用户故事**：研究员在 arXiv 上看到一篇论文，点击 Neuro Connector，自动保存 PDF 和元数据到"神经电极"Collection

## F2 Knowledge Tree（知识树）

- **目标**：实现 Project → Collection → Knowledge Unit 的层级浏览
- **核心流程**：用户在左栏浏览 Knowledge Tree，点击 Knowledge Unit 在 Workspace 中打开
- **用户故事**：用户在 Knowledge Tree 中找到"类器官培养"下的所有论文

## F3 Knowledge Unit 管理

- **目标**：Knowledge Unit 的创建、编辑、删除、移动
- **核心流程**：用户选择 Template → 创建 Knowledge Unit → AI 自动初始化 Knowledge.md
- **用户故事**：用户手动上传 PDF，系统自动创建 Paper Knowledge Unit 并生成初始 Knowledge.md

## F4 AI Engine

- **目标**：围绕 Knowledge Unit 提供 AI 理解、提取、标注、推理
- **核心流程**：用户选择 Knowledge Unit → 触发 AI 分析 → AI 阅读 Knowledge.md → AI 更新 Knowledge.md
- **用户故事**：用户导入一篇新论文，AI 自动提取关键参数（电极材料、阻抗值）写入 Knowledge.md

## F5 AI Chat

- **目标**：在 AI Panel 中实现基于当前项目上下文的自然语言对话
- **核心流程**：用户提问 → AI 优先检索当前 Project 的 Knowledge.md → AI 结合联网搜索 → 生成回答
- **用户故事**：用户提问"这个项目中阻抗最低的电极材料是什么？"，AI 检索所有论文的 Knowledge.md 并给出对比表格

## F6 View System

- **目标**：提供 Markdown / Graph / Timeline / Kanban / Calendar 等多种视图
- **核心流程**：用户切换 View 类型 → Workspace 主体切换为对应视图
- **用户故事**：用户切换到 Graph View，看到所有论文与实验之间的关联网络

## F7 Follow（知识演化）

- **目标**：AI 持续跟踪项目进展，自动更新 Knowledge.md 和任务状态
- **核心流程**：用户向 AI 汇报进度 → AI 更新 Knowledge.md 标签 → Graph/View 自动同步更新
- **用户故事**：用户完成当天实验后告诉 AI"阻抗测试完成，结果为 5kΩ"，AI 更新实验 Knowledge.md，同步更新 Graph 和排名表

## F8 Plugin System

- **目标**：开放的插件架构，允许社区扩展
- **核心流程**：用户安装 Plugin → Plugin 注册到系统 → 用户使用 Plugin 功能
- **用户故事**：开发者创建了"自动生成综述报告"插件，用户在 AI Panel 中一键生成文献综述

# 第四部分：技术架构概要

## 数据存储

| 层级 | 技术方案 | 说明 |
|------|---------|------|
| 文件系统 | 本地目录 + 加密云盘（可选） | 存储附件（PDF/图片/视频） |
| 关系型数据库 | SQLite | 存储 Knowledge Unit 元数据、Relation 关系 |
| 向量数据库 | FAISS（可选） | AI 检索增强（RAG） |
| Knowledge.md | Markdown 文件 | AI 可读/写的人类可读知识索引 |

## 前端技术

- 桌面应用：Electron（跨平台）
- UI 框架：React / Vue（待定）
- 视图渲染：内置 Markdown 解析器 + D3.js（Graph 视图）+ 第三方图表库

## 后端技术

- 本地服务：Node.js / Python（可选）
- AI 接口：Claude API / OpenAI API / 国产大模型 API
- 插件系统：Plugin SDK + MCP 协议

# 附录：原始 1.0 概念升级对照表

| 1.0 原始概念 | 最终定位 | 升级说明 |
|------------|---------|---------|
| Neuro connector 浏览器插件 | Connector Plugin | 从功能降级为插件，以保持系统核心的纯净 |
| Nuero 项目 / Entry 文件夹 | Project → Collection → Knowledge Unit | 用 Knowledge Unit 替代"文件夹"概念 |
| Entry 模式（PDF阅读器/翻译器等） | Knowledge Workspace 集成工具 | Entry 只是生命周期的一个阶段，不是独立模式 |
| 主节点 markdown 文件 | Knowledge.md（AI Knowledge Index） | 去掉"主节点"名称，强调 AI 索引而非文档 |
| Thinking 模式 / AI 打标 | AI Engine（Understand / Extract / Tag / Reason） | Thinking 太抽象，统一为 AI 引擎能力矩阵 |
| Print 模式 / 可视化 | View System（Graph/Timeline/Kanban/Calendar 等） | Print 命名不准确，统一为 View |
| Follow 模式 | Knowledge Cultivation（知识演化） | Follow 太被动，改为"培育"——知识在持续生长 |
| 项目管理（任务模板） | Task Template（Knowledge Unit 的一种类型） | 任务与论文、实验平级，都是 Knowledge Unit |
| 标签/打标 | AI 自动 Tag + Relation 关联 | 标签服务于可视化和 Graph |
| 数据排名表 | AI Engine 的 Reason 能力 + Table View | 排名是 AI 推理结果的一种呈现 |
| 插件接口 | Plugin Architecture（Connector/Zotero/Obsidian 都是插件） | 所有外部集成统一为插件 |
| 开源社区接口 | Plugin SDK + API + MCP | 体系化开放，降低开发门槛 |
